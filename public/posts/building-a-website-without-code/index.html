Building a website without code

Build and publish a website using only Markdown and YAML files. No code required.

Building a website without code

Context

  • Motivation for this project (thought experiment)
  • Secure by design, reducing the no. of attack vectors to exploit
  • What this project is not, when NOT to use it

Requirements Checklist

  • Architecture should be simple & intuitive, so that anybody can use it
  • We will use a git repo as source of truth, to store our site content
  • Only markdown (.md) and YAML (.yaml) files will be allowed in git
    • Markdown is used for page contents (blog, about, contact us)
    • YAML can be used for website config, metadata and a data store
    • Only exception is git metadata (eg: .gitignore, .gitmodules)
  • Ignore everything else (eg: added to .gitignore), just noise
    • Anything not stored in git is considered transient, temporary
    • Focus on what is important, your content and data
  • Solution should be extendable, to allow more advanced use cases
    • Look and feel should be customisable, using (third party) themes/css
    • Support going from a static website to a dynamic web application / SPA
    • Server-side functionality should be supported (eg: if you need a backend)
    • Cloud native friendly, capable of deploying to Kubernetes / Docker
  • Follow the principals and best practices of a 12 Factor Application

Solution Design and Architecture

  • The ideal solution in the simplest form:
    • content/
    • data/
    • config.yaml
    • Generated site (public/) published using any static web server
  • Separation of content, presentation and data, using filesystem as storage
    • Simple to maintain, change and backup files and folders, git as source of truth
    • Advanced use cases should support self-hosting / private cloud storage
    • Data is never directly exposed to the internet, stays private to you
  • Supporting advanced workflows:
    • Static workflow - Building a custom website / theme and publish it yourself
    • Dependency workflow - Pull resources from third party dependencies (eg: theme)
    • Backend server workflow - Add dynamic content served from backend server
    • Cloud-Native workflow - Deploy and scale to any size, using Kubernetes
  • Security considerations, and best practices
    • Use a trusted hosting partner, avoid publicly exposed cloud storage buckets
    • Treat data as sensitive and restricted, avoid leakage and exposure
    • Clearly separate backend server traffic from static website traffic
    • Add a gateway-level authentication proxy, avoid client-side authentication
    • Admin pages (eg: control panel) should be restricted and separately hosted
